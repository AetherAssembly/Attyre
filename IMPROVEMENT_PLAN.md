# Attyre: Improvement & Desktop Migration Plan

> **Scope:** No code changes are made here. This is a planning document only.
> **Goal:** Ship a real laptop app while keeping Attyre alive on mobile, and fix the recurring mobile navigation bug for good.

---

## Where Things Stand

The current stack is a vanilla JS single-page app with hash-based routing, all data in `localStorage` (compressed with LZString), and images stored as base64 URIs. There is no bundler and no framework — just ES modules served directly. Deployment is Cloudflare Workers/Assets via Wrangler.

This is actually a strong foundation. The lack of a framework means there is almost no rewrite cost involved in wrapping it in a desktop shell, and the local-first data model is perfect for a native app.

---

## Phase 0: Fix the Mobile Bug (Before Anything Else)

The recurring mobile issue is: you can't leave the home page, and the home page content is static. The changelog shows this has been patched multiple times without fully resolving it.

### Root Cause Analysis

After reading the code, there are three likely culprits working in combination:

**1. Stale or missing service worker.**
There is no `service-worker.js` in the repo. That means any previously deployed service worker from an earlier version is still sitting in users' browsers, intercepting fetches and serving cached old builds. When someone installs the PWA, their browser caches that old worker. New deployments don't automatically purge it because there is no `skipWaiting` / `clients.claim` logic to push updates out.

**2. The opacity transition in `renderPage` can silently fail.**
In `app.js`, the router does `app.style.opacity = '0'`, then waits 120ms, then renders, then sets `opacity = '1'`. If any render function throws an error (like referencing a CSS class that doesn't exist in `style.css`, which the changelog shows has happened multiple times), the `opacity` never gets set back to `1`. The page renders with invisible content. On mobile the errors are harder to see because there is no DevTools open. This has already caused multiple "static page" complaints.

**3. Portrait orientation lock in the manifest.**
`manifest.json` has `"orientation": "portrait-primary"`. This is cosmetically fine but contributes to the app feeling broken and locked down.

### The Fix (Plan, Not Implementation)

- Add a minimal service worker with a network-first strategy for navigation and a versioned cache name. On deploy, bump the cache version to force old workers to update. Include `skipWaiting()` and `clients.claim()` so the new worker takes over immediately.
- Wrap every page render function call in `app.js`'s router in a `try/catch`. If a page throws, render a visible error card inside `#app` and set `opacity = '1'` so the user at least sees something. Right now errors are swallowed silently and the fade-out stays.
- Change `"orientation"` to `"any"` in the manifest.
- Add a `cache-busting` meta tag or version query param to ensure fresh assets after deploys.

## Phase 1: Desktop App via Tauri

### Why Tauri and Not Electron

Electron bundles its own copy of Chromium. A typical Electron app is 80-120 MB even for something this small. Tauri uses the OS's built-in webview (WebKit on Mac, WebView2 on Windows, WebKitGTK on Linux), so the final binary is roughly 8-15 MB.

More importantly, the existing frontend code — every `.js` file, `index.html`, and `style.css` — requires **zero changes** to run in Tauri's webview. Tauri just wraps it. The dev workflow stays almost identical.

Tauri 2.0 (current stable) also supports Android and iOS build targets, meaning when there's eventually budget for a Play Store listing or Apple Developer account, the mobile app becomes a build flag rather than a rewrite.

### What Tauri Actually Adds

Tauri adds a thin Rust shell (`src-tauri/`) that handles the native window, OS integration, and app lifecycle. You do not need to write Rust to get basic functionality — the scaffolding is generated by `tauri init` and the defaults work out of the box.

The web layer talks to Tauri via a JavaScript API (`@tauri-apps/api`) for anything that needs OS access: file system, notifications, system tray, and so on. For Attyre's current feature set, you may not need any of this initially since everything is already self-contained in `localStorage`.

### What Changes in the Repo

```bash
Attyre/
  src-tauri/          ← new (Rust shell, generated by tauri init)
    Cargo.toml
    tauri.conf.json   ← configures window size, title, icons, etc.
    src/
      main.rs         ← boilerplate entry point
  package.json        ← add @tauri-apps/cli as devDependency, add tauri scripts
  index.html          ← no changes needed
  js/                 ← no changes needed
  style.css           ← no changes needed
```

That's it for the initial migration. Everything else is unchanged.

### Build & Distribution

Tauri's `tauri build` produces:

- `.dmg` + `.app` for macOS
- `.msi` + `.exe` (NSIS installer) for Windows
- `.AppImage` + `.deb` for Linux

Distribution is via GitHub Releases, which is already set up and free. Users download the right installer for their platform. No app store, no signing fees for Linux and Windows (macOS does eventually need a developer certificate to avoid Gatekeeper warnings, but it is not required for early users who can allow it via System Preferences).

### Dev Workflow Change

Currently: `npm run dev` runs `python3 -m http.server 8000`.

With Tauri: `npm run tauri dev` opens a native window pointing at a local dev server that Tauri spins up automatically. Hot reload continues to work the same way. The existing `dev` script stays as a fallback for browser testing.

## Phase 2: Storage Improvements (Medium Term)

### The LocalStorage Problem

Right now, images are stored as base64 data URIs inside `localStorage`. This causes two issues: (1) `localStorage` has a 5-10 MB cap per origin, which is easy to hit with a wardrobe full of photos, and (2) the data is tied to a specific browser profile, not the OS user account. On a laptop app, users expect their data to just be there, not locked inside a browser sandbox.

### The Plan

Once Tauri is in place, migrate image storage to the filesystem using Tauri's `fs` plugin. Images get saved to `$APPDATA/attyre/images/` and referenced by filename instead of base64 URI. The `store.js` `saveItems` and `getItems` functions get updated to store a filename string in `imageUri` instead of a full base64 blob.

For the rest of the data (items, outfits, calendar), switch from raw `localStorage` to `tauri-plugin-store`, which is a Tauri-maintained plugin that provides a JSON file-backed store at a proper OS data path. The API is nearly identical to `localStorage` so the `store.js` layer changes are minimal.

This also solves the "Your data is stored locally in this browser. Export a backup before clearing your cache" warning on the home page — that warning becomes irrelevant for desktop users because the data is no longer browser-ephemeral.

## Phase 3: Keep Mobile Alive (Web Version)

The Cloudflare deployment stays running. Mobile users continue to use the web version (or install it as a PWA with the fixes from Phase 0).

The web and desktop versions share the exact same codebase for everything under `js/`. The only conditional behavior is in `store.js`, where a check for `window.__TAURI__` (Tauri injects this global) can switch between the localStorage path and the Tauri plugin path. Non-Tauri environments (mobile browser, desktop browser) continue using `localStorage` exactly as they do now.

This is a clean separation and requires no build-time flags or separate branches.

### Future Mobile Native (When Budget Allows)

When there are users or revenue to justify it:

- **Android:** Tauri 2.0 can build an `.apk` today. APK sideloading is free. A Play Store listing costs a one-time $25 fee.
- **iOS:** Requires an Apple Developer account ($99/year). The Tauri iOS target exists in 2.0, so it would be a build config change, not a rewrite.

## Phase 4: Code Health (Ongoing)

These are improvements that don't need to happen before the desktop migration but should happen alongside it.

### Add a Build Step (Vite)

Currently the dev server is `python3 -m http.server`. This works but means: no hot module replacement, no tree-shaking, no TypeScript, no path aliases. Tauri includes a Vite integration by default when you run `tauri init`, so this comes for free as part of the migration.

### Fix the Recurring CSS Class Bug Pattern

Several changelogs show the same class of bug: a page file references a CSS class that doesn't exist in `style.css`. The fix is to add a CSS linting step (stylelint + a custom rule, or just a grep in CI) that fails the build if any class name used in JS is not defined in the stylesheet. This would have caught every one of these regressions before they shipped.

### Tighten the Error Boundaries

As noted in Phase 0, render functions that throw silently break the app. Every `render*` function should have a standard error boundary pattern so failures are visible and recoverable.

### Remove Deprecated Functions

`store.js` has `isColorblind()` and `setColorblind()` marked `@deprecated` since they were replaced by `isAccessibilityMode()` / `setAccessibilityMode()`. These should be removed once a search confirms nothing still calls them.

---

## Summary: Order of Operations

1. **Fix the mobile bug** (Phase 0) — independent of everything else, unblocks mobile users now.
2. **Tauri migration** (Phase 1) — wraps existing code, minimal disruption, produces a real desktop app.
3. **Storage migration** (Phase 2) — follow-on once desktop is stable, improves reliability significantly.
4. **Keep web version running** (Phase 3) — no work required, just don't remove the Cloudflare deployment.
5. **Code health** (Phase 4) — ongoing alongside everything else.

The total scope of the Tauri migration (Phase 1 alone) is roughly: run `tauri init`, configure `tauri.conf.json` with the right window settings and icons, wire up the `tauri dev` and `tauri build` npm scripts, and test. The existing application code does not need to change for the first release.
